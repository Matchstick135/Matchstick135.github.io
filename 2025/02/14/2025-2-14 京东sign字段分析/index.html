京东sign字段分析

发表于2025-02-14|逆向日记
|总字数:4.3k

这么说来，自从学会抓包的基本流程后，似乎一直都是在单纯为了抓包而抓包
想到这里才如梦初醒，抓包不是一直以来都是逆向中极为重要的一环吗，自己却如此割裂地看待两者，属实是浅薄了
于是，为了让自己进行的抓包操作，不再是”单纯为了抓包而抓包”，于是便有了这篇记录

抓包

打开Charles，再配置系统代理后 即可开始抓包

锁定如下URL，分析其参数

可以看到 sign字段处于加密状态，通过长度32的Hex字符串特征，初步猜测是MD5哈希值

java层分析

frida检测绕过

frida附加时，应用意外退出
因此先从改文件名、指定端口开始尝试

没想到直接就能成功附加了，省了很多事

加密入口定位

一般有如下三种思路，这里选择第三种

  1. “sign”关键词搜索
  2. hook StringBuilder, hashMap
  3. hook getBytes

哈希算法通常先使用 java.lang.String.getBytes 方法将字符串转换为字节数组
且sign字段的生成，通常会传入其他常规字段的值 参与计算
由此为切入点，hook getBytes多个重载方法，当传参包含上述包中一系列字段名时，打印调用栈

虽然存在多次调用，但发现调用栈都遵循如下结构

看来加密入口大概率就是位于这条调用链中了

先hook genSignJDGuardHelper$1.genSign，打印传参

发现其传入的URI中已包含sign字段

看来字段生成逻辑 还要位于更上层调用链

沿着调用链往上hook HttpSettingTool.doSignUsingJdGuard

仍然包含sign字段

继续往上，来到 ParamBuilderForJDMall.setupParams
发现其在为 HttpRequest 对象添加各种自定义参数，进而构建 HTTP请求

而调用的 signature方法，正是在生成签名信息，由此深入追踪

其中从 HttpSetting 对象及工具类中获取 functionId, versionName等信息
而后与 str（请求体参数）, str2（UUID）一同传入 signature(Context context, String str, String str2, String str3, String str4, String str5)

然而跟进该方法发现是一个接口，仅通过查找引用是无法来到其实现类的

不过接口在实现时 肯定会再次提及其方法签名
基于此思路，我们可以直接对方法签名进行全局搜索

由此来到 BitmapkitUtils类

看来sign字段的生成，底层由native函数 getSignFromJni实现
且疑似来自于 运行期间动态加载的”jdbitmapkit”模块

一般还是通过遍历各模块导出符号 来定位native函数所在

不过这里既然已有明确方向，所以先选择解包查看 libjdbitmapkit.so导出表
成功搜到静态注册的 getSignFromJni

进入native层前先提一嘴
后续分析过程中肯定涉及 BitmapkitUtils.getSignFromJni的多次调用，然而每次变化的参数会对分析产生干扰
因此这里选择 固定传参+主动调用，触发每次的 BitmapkitUtils.getSignFromJni

native层分析

主函数

跟进到 sub_127E4

其中逻辑大致可分为以下几部分

  1. 先使用 StringBuffer对传入的 functionId, body, uuid等一系列参数进行拼接

  2. 再基于 gettimeofday返回的时间 生成时间戳，作为st字段值

  3. 再基于 lrand48生成的两个随机数（0/1/2），与”1”拼接，作为sv字段值

  4. 然后将上述随机数，与通过 getBytes转换成字节数组的 StringBuffer，一同传入encrypt1

  5. 动态分配内存，成功则进入if分支
    此时将 encrypt1结果依次传入 encrypt2, encrypt3进行加密，转换成Hex字符串，作为sign字段值

  6. 最后依次将 st, sign, sv字段值拼入 StringBuffer，而后返回字符串至java层

  7. 内存分配失败，则进入else分支
    此时 sign, sv字段值会被统一固定，而后返回字符串至java层

加密函数 I

回过头来看前面提到的 encrypt1，传入了 初期StringBuffer，以及组成sv字段的三个int

其中先基于随机数二，生成乱序数组（0/1/2）
再基于随机数一，从该数组中选取 后续要进入的加密case（0/1/2），以及要与字节数组一同传入加密的 32字节key

开始各case分析前先提一嘴，因为其选取是随机的，即便一开始固定了java层传参，后续hook打印信息时 还是会存在拦截不到的情况
因此这里在前面 主动调用脚本的基础上添加功能
仅在 BitmapkitUtils.getSignFromJni执行期间，拦截 gettimeofday, lrand48，固定其返回值

整除字节处理

目光转至 sub_10EA4，数据被分为8字节一组传入
从Graph view可以看出是一个非常冗长的函数，其中逻辑大致分为三部分

  1. 使用 0x01,0x02,0x04,0x08,0x10,0x20,0x40,0x80，分别对传入的各字节的每个位 进行提取、存储
  2. 每次循环，使用 0x01-0x80 对传入key的每个位 进行提取，并基于其与0比较的结果，进入16个不同分支
    每个分支中，都在对前面存储的位数据 进行交换打乱
  3. 最后还是使用 0x01-0x80，对打乱的位数据 进行重组复原

尝试过对着伪代码 手动还原出上述算法，然而成品实在是过于丑陋
又臭又长，根本称不上真正意义上的”还原”，就不放出来了

这里参考这位大佬的思路
如果单纯把 sub_10EA4视作黑盒的话，其无非只是将传入的64位数据 进行重排序
也就是说，每一位数 在输入至输出间 都存在明确的映射关系

因此我们可以利用差分攻击的思想，依次将 2^0, 2^1, 2^2… 2^63 传入sub_10EA4
然后捕获每次输出中”1”所在的位，即可得到各个位 前后映射关系了

这里选择hook sub_10EA4，依次将传入的8字节数据改成 2^0, 2^i… 2^63
然后每次将输出的log2(x) 存入数组的索引 i 处

由此得到映射表

基于此映射表，就可以十分优雅地写出 真正意义上的”还原”算法了

末尾字节处理

但事情远没有这么简单，sub_10EA4 只是对8字节整除部分进行加密
对于末尾不足8字节的数据，还得进行额外处理

  1. sub_10E80中，sub_10DD0用于创建对象，其首8字节为指向 off_16E38处虚函数表的指针
  2. sub_10E80后，通过指针调用虚表首个函数 sub_10D70，即来到 末尾数据的加密逻辑处
    其中基于剩余的字节个数（1-7），进入7个不同分支

    每个分支对应一套独立算法，人麻了……

这里就挑理论上最简单的 sub_4B7C（加密剩余1个字节）上手了
整体逻辑 和前面处理整除字节相似，大致分为三部分
先是位提取、存储逻辑；再是基于key，但貌似不是单纯的位交换打乱；最后重组复原

先试着沿用前面 差分攻击的思想，依次将 2^0, 2^1, 2^2… 2^7 传入sub_4B7C，然而对比输入和输出的二进制特征
然而却得到如下映射，一下看不出所以然

由此控制变量，把每次输出与 传入0时的输出比对
豁然开朗，原来 位在映射期间还进行了 条件性取反操作

基于此映射规则，再次优雅地写出”还原”算法

至于剩下的6个分支，大概率也是按照上述套路

总结

  1. 入口方法：
    com.jingdong.common.utils.BitmapkitUtils.getSignFromJni
  2. libjdbitmapkit.so中 先后经过三个加密函数
  3. 第一个加密：
    三套加密方案，随机选取。
    前两套每次加密8字节，末尾字节有七套加密方案，基于剩余字节数选取。加密类型为位置换；
    后一套使用简易的自定义加密。加密类型为流加密
  4. 后两个加密：
    分别为标准 Base64, MD5
